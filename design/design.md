# llm-judge-summary-agent 設計メモ

この文書は「なぜこの設計にしたか」の記録です。使い方は README、概念入門は 01_airepo の `00_入門_…` ／ `00_詳説_…` を参照。

## 実証する設計パターン
**LLM-as-Judge ＋ 決定的ゲート**。決まった正解（golden）が無い出力（要約・分類・言い換えなど）を、
LLM の審査員がルーブリックで採点し、その点数を固定ルールで合否に変換する。

## なぜ「LLM-as-Judge」を選んだか（他方式との比較）★この repo の肝
- 要約には**「唯一の正解」がありません**。良い要約は何通りもある。
  → だから **golden 比較（差分テスト・静的 golden）が原理的に使えない**（比べる正解が1つに決まらない）。
- **メタモルフィック**も弱い：「良い要約なら必ず満たす crisp（明確）な関係」を作るのが難しい（良し悪しが文脈依存）。
- 残る現実的な手は、**人間の評価を LLM で代理する（LLM-as-Judge）**こと。完璧ではないが、正解の無い領域を測れる。
- 対比：js-to-ts＝差分テスト、py2to3＝静的 golden、metamorphic-sort＝性質。本 repo は「**正解も性質も作れない、質が主観的な出力**」の列を担当する。

## 設計の肝：採点（LLM）と判定（決定的）を分ける
- 採点 ＝ judge.md（LLM）。source と candidate を読み、忠実性・網羅性・簡潔性を 0〜5 ＋捏造真偽で採点 → JSON。
- 判定 ＝ oracle.py（決定的）。採点 JSON を、明示した合否ルールに通す（同じ採点なら必ず同じ合否）。
  → LLM の曖昧さを「採点」に閉じ込め、合否は再現可能にする。

## 合否ルール
hallucination == false かつ faithfulness>=4 かつ coverage>=3 かつ conciseness>=3。

## 信頼性（オラクル検証）
- 判定ルール … oracle.py --selftest が、良い採点→PASS・悪い採点→FAIL を確認（完全に決定的）。
- 審査員 … 各 corpus に good（正例）と bad_halluc（捏造）／ bad_missing（要点欠落）を置き、
  judge が good を高得点（PASS）・bad を低得点（FAIL）に分けられるかで妥当性を見る（ラベルを伏せたブラインド採点）。

## 非決定性・バイアスへの構え
LLM 採点は揺れ、バイアス（長い回答を高評価する等）も持つ。アンカー付き 0〜5 ＋明確な合否ルールで揺れを抑える。
さらに固めるなら、judge を複数回／複数観点で走らせて多数決にする（拡張余地）。

## corpus（クリーンルーム・一般題材）
- 01_photosynthesis … 光合成のしくみ
- 02_bicycle_gear … 自転車の変速機

## 既知の限界
- 審査員も LLM なので、絶対の物差しではない（重要判断は人手評価と併用が理想）。
- ルーブリックと閾値の設計が質を左右する。
